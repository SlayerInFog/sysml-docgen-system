1.2　 系统概述
　　基于SysML模型的文档自动生成系统主要面向系统建模与工程文档编制场景，适用于以SysML模型为核心开展需求分析、系统设计和文档管理的项目开发过程。系统的用途是通过对SysML模型进行统一管理、组织和复用，减少传统人工编写文档过程中存在的信息重复录入、文档更新滞后以及模型与文档内容不一致等问题，提高工程文档编制效率和项目协同能力。
　　本软件的用途是为用户提供一个基于浏览器访问的模型驱动文档生成平台。用户可以在系统中完成模型导入、模型管理、模型内容查看、模型轻量编辑、模板选择、文档自动生成、文档预览与导出等操作，从而实现从模型到文档的一体化处理过程。软件采用B/S架构，运行于Windows平台，前端采用Vue技术实现用户交互界面，后端采用Python实现业务逻辑与接口服务，数据库采用MySQL保存系统运行过程中涉及的用户信息、项目数据、模型数据、模板数据以及文档生成记录等内容。
　　本系统具有以下一般特性：以SysML模型作为核心数据来源；支持模型文件导入、解析和统一管理；支持在Web端对模型内容进行浏览和轻量编辑；支持按照预定义模板自动生成工程文档；支持生成文档的预览、导出和历史记录管理；支持多用户在权限控制下进行协同使用。系统整体采用前后端分离方式设计，具有较好的可扩展性、可维护性和跨终端访问能力。
　　本系统为软件工程课程综合设计项目，当前处于课程设计开发阶段。系统开发环境为Windows操作系统，采用前后端分离开发模式，由项目组成员协同完成需求分析、系统设计、编码实现、测试和文档整理等工作。系统当前主要运行于本地开发与测试环境，后续计划部署在Windows服务器或实验室环境中，供相关用户通过浏览器访问使用。系统运行维护工作主要包括用户管理、数据备份、日志管理、系统配置维护及异常处理等内容。
　　基于SysML模型的文档自动生成系统软件的功能包括模型管理、视图编辑、模型工具接入、文档生成管理、用户权限管理和日志管理等功能。软件系统由模型管理模块、视图编辑器模块、模型工具接入模块、文档生成管理模块和公共支撑模块组成。
　　本软件属于II类C级软件。
a) 需方：南京航空航天大学；
b) 用户：课程设计项目小组、教师；
c) 开发方：课程设计项目小组；
　　本系统当前运行现场为Windows操作系统下的本地开发与测试环境，开发人员通过本机完成前端、后端和数据库的部署与调试。计划运行现场为Windows操作系统支持的教学实验室、局域网或校园网环境，用户通过浏览器访问系统，无需安装专用客户端软件。

3　 软件体系架构
　　本章从软件总体结构、分层架构、子系统组成、软件目录映射、主要业务数据流和运行部署关系等方面，对基于 SysML 模型的文档自动生成系统的软件体系架构进行详细说明。系统采用 B/S 架构和前后端分离设计方式，用户通过浏览器访问系统前端页面，前端通过统一接口与后端服务进行数据交互，后端负责业务逻辑处理、模型解析、文档生成、权限校验、日志记录和数据访问，数据库及文件存储目录负责保存系统运行过程中产生的结构化数据和非结构化文件。
3.1　 总体架构
　　基于 SysML 模型的文档自动生成系统采用浏览器/服务器架构。系统整体由用户终端、前端应用、后端应用、数据库和文件存储五个部分组成。
　　用户终端主要是安装 Chrome、Edge 等主流浏览器的普通计算机。用户不需要安装专用客户端软件，只需通过浏览器访问系统前端页面，即可完成用户登录、项目管理、模型上传、模型查看、模型轻量编辑、模板管理、文档生成、文档预览和文档导出等操作。
　　前端应用采用 Vue 技术栈实现，主要负责页面展示、路由跳转、用户输入、表单校验、状态管理和接口调用。前端通过 HTTP 请求向后端提交业务数据，并接收后端返回的 JSON 数据、文件下载响应或操作提示信息。
　　后端应用采用 Python 技术栈实现，主要负责接口服务、业务处理、权限校验、模型文件解析、文档内容生成、文件保存、日志记录和数据库访问。后端对外提供统一的 Web API 接口，前端所有业务操作均通过这些接口完成。
　　数据库采用 MySQL 保存系统的结构化业务数据，包括用户信息、角色信息、项目信息、模型元数据、模型元素、模型关系、模板信息、文档生成记录和操作日志等。文件存储目录用于保存用户上传的模型源文件以及系统生成的 HTML、DOCX、PDF 等文档文件。
　　系统总体架构如图 3-1 所示。
　
图3-1系统总体架构图
　　图 3-1 描述了系统从用户访问到业务处理再到数据持久化的整体关系。前端应用是用户操作入口，后端应用是业务逻辑处理中心，数据库和文件存储目录共同构成系统的数据支撑基础。系统通过这种结构实现界面展示、业务处理和数据存储之间的分离，降低系统耦合度，提高后续维护和扩展的便利性。
3.2　 软件分层结构
　　从软件逻辑结构角度看，本系统划分为用户访问层、前端表示层、后端接口层、业务服务层、数据访问层和数据存储层。各层之间按照自上而下的方式进行调用，上层负责发起请求和组织业务流程，下层负责提供数据、服务和基础能力。

3.3　 前后端结构
　　本系统采用前后端分离结构。前端和后端分别作为相对独立的软件部分进行开发、运行和维护，二者通过 HTTP 接口进行通信。
　　前端部分主要包括页面视图、路由配置、状态管理和接口封装。页面视图用于展示登录页、首页、项目管理页、模型管理页、模板管理页、文档生成页和后台管理页等；路由配置用于控制不同页面之间的跳转关系；状态管理用于保存当前登录用户、认证令牌、角色权限和部分全局状态；接口封装用于统一处理与后端 API 的通信。
　　后端部分主要包括接口路由、核心配置、数据模型、数据结构和业务服务。接口路由负责提供认证、项目、模型、文档和日志相关接口；核心配置负责数据库连接、系统配置和安全认证；数据模型负责定义用户、项目、模型、文档和日志等数据库对象；数据结构负责对接口输入输出数据进行规范化约束；业务服务负责完成模型解析、文档生成和审计日志记录等核心业务操作。
　　前后端之间的数据交互主要包括三类：第一类是普通业务数据交互，即前端以 JSON 格式提交登录信息、项目数据、模型元素编辑信息、模板配置和文档生成参数；第二类是文件上传交互，即用户上传 SysML、UML、XMI、XML 或 JSON 模型文件时，前端以文件上传方式提交给后端；第三类是文件下载交互，即用户导出 HTML、DOCX 或 PDF 文档时，后端以文件响应形式返回生成结果。

　　后端软件采用接口层、核心支撑层、数据模型层、数据结构层和业务服务层相结合的组织方式。其主要目录及职责如下表所示。

3.7　 主要业务数据流
　　本系统的核心业务数据流围绕"用户登录-项目管理-模型导入-模型解析-模型查看与编辑-模板管理-文档生成-文档导出-日志记录"展开。
　　用户首先通过登录页面提交账号和密码。前端将登录信息发送至后端认证接口，后端完成用户身份校验，校验成功后生成认证令牌并返回前端。前端保存认证令牌，并在后续请求中携带该令牌，以便后端完成权限验证。
　　用户进入项目管理页面后，可以创建或选择项目。项目数据保存到数据库中，并作为模型、模板和文档数据的组织依据。用户在模型管理页面选择所属项目并上传模型文件，后端接收文件后进行格式校验和保存，然后调用模型解析服务提取模型元素、模型关系、元素名称、元素类型和描述信息，并将解析结果写入数据库。
　　模型解析完成后，用户可以在模型管理或模型视图页面查看模型元素和模型关系。对于具备编辑权限的用户，系统允许其修改模型元素说明、备注等轻量信息。修改后的内容通过前端接口提交到后端，并更新数据库中的模型元素记录。
　　用户在模板管理页面创建或维护文档模板。模板数据保存后，可在文档生成页面被选择使用。用户选择项目、模型和模板后，前端向后端提交文档生成请求。后端文档生成服务读取对应模型数据和模板内容，将模型信息填充到模板中，生成 HTML、DOCX 或 PDF 等格式的文档文件，同时在数据库中保存文档生成记录。
　　用户可在线预览生成结果，也可将生成文档导出到本地。系统在用户登录、项目维护、模型上传、模型解析、模板维护、文档生成和文档导出等关键环节记录操作日志，用于后续审计、问题定位和系统维护。
3.8　 运行部署结构
　　课程设计阶段，本系统可采用本地一体化部署方式，即前端服务、后端服务、数据库服务和文件存储目录部署在同一台 Windows 开发计算机上。该部署方式便于项目组进行开发调试、接口联调、演示准备和课程验收。
　　在本地开发运行时，前端应用通过开发服务器启动，用户通过浏览器访问前端页面；后端应用通过 Python 服务启动，对外提供 API 接口；MySQL 数据库在本机或局域网服务器中运行，用于保存结构化业务数据；上传模型文件和生成文档文件保存在后端指定文件目录中。
　　后续如部署到教学实验室或服务器环境中，可将前端应用、后端应用和数据库服务分别部署在不同主机上。前端可作为静态资源部署，后端作为接口服务部署，数据库作为独立数据服务部署，文件存储目录可部署在后端服务器本地目录或统一文件存储位置。无论采用本地部署还是服务器部署，系统的前后端接口关系、数据访问方式和文件管理方式均保持一致。
3.9　 架构设计特点
　　本系统的软件体系架构具有以下特点：
　　前后端分离。系统前端负责用户交互和页面展示，后端负责业务逻辑和数据访问，二者通过统一接口进行通信，有利于降低界面层与业务层之间的耦合。
　　分层清晰。系统按照表示层、接口层、业务服务层、数据访问层和数据存储层组织，每一层职责相对明确，便于开发人员理解、实现和维护。
　　模块化设计。系统按照功能职责划分为多个子系统，各子系统之间通过接口和数据对象协同工作，有利于后续扩展模型格式、模板类型、文档导出格式和权限控制能力。
　　模型驱动。系统以 SysML 模型数据为核心输入，将模型解析结果作为文档生成的重要数据来源，减少人工重复录入，提高模型数据与工程文档之间的一致性。
　　可追溯性较强。系统保存项目、模型、模板、生成文档和操作日志等信息，能够记录文档生成所依赖的模型数据和模板来源，为后续维护、审计和问题定位提供依据。
　　适合课程设计实现。系统架构在满足主要功能需求的基础上，避免引入过于复杂的分布式部署、高并发处理和复杂权限体系，符合当前课程设计开发周期、演示要求和小组协同开发条件。
4　 CSCI详细设计
4.1　 公共支撑软件单元 SYSMLDOCGEN-CSU-SUPPORT
4.1.1　  功能描述
　　公共支撑软件单元的项目唯一标识为 SYSMLDOCGEN-CSU-SUPPORT。该软件单元是基于 SysML 模型的文档自动生成系统的基础支撑单元，主要为项目管理、模型管理、视图编辑、模板管理、文档生成、文件管理等业务软件单元提供统一的用户认证、身份识别、角色权限控制、用户管理和日志审计能力。
　　本软件单元在系统运行过程中不直接完成模型解析或文档生成等核心业务处理，但它是所有受保护业务功能正常运行的前置条件。用户访问系统前，应先通过公共支撑软件单元完成登录认证；用户访问项目、模型、模板、文档和日志等业务接口时，系统应通过本软件单元对用户身份和角色权限进行校验；用户完成注册、登录、用户信息维护、模型上传、文档生成等关键操作后，系统应调用日志审计功能记录操作信息。
　　本软件单元由以下子功能组成：
a) 用户注册功能：接收用户注册信息，校验用户名、邮箱、密码和角色字段，创建用户记录，并保存加密后的密码信息；
b) 用户登录功能：接收用户名和密码，查询用户记录，校验密码是否正确，判断用户是否处于启用状态，并在认证通过后生成访问令牌；
c) 当前用户识别功能：根据前端请求携带的认证令牌解析当前用户身份，并向其他业务单元提供当前用户对象；
d) 角色权限控制功能：根据当前用户角色判断其是否具有访问指定接口或执行指定操作的权限；
e) 用户管理功能：支持管理员查看用户列表、启用或停用用户、调整用户角色等操作；
f) 日志审计功能：记录用户注册、登录、用户信息维护以及其他业务单元调用时产生的关键操作日志。
　　当前源码中，认证接口主要位于 backend/app/api/auth.py，安全认证逻辑主要位于 backend/app/core/security.py，用户数据模型位于 backend/app/models/user.py，日志数据模型位于 backend/app/models/audit.py。其中 auth.py 已包含注册、登录、当前用户、用户列表、用户选项和用户更新等接口逻辑；security.py 已包含密码哈希、密码校验、JWT 生成、当前用户识别和角色校验等支撑函数；user.py 和 audit.py 分别定义用户表和审计日志表的数据结构。
　　本软件单元的程序特点如下：
a) 运行方式：随系统后端服务启动而加载，属于 Web 后端服务中的基础功能单元；
b) 处理方式：以请求驱动方式运行，当前端发起登录、注册、用户管理或受保护接口请求时触发执行；
c) 处理性质：主要为同步处理，认证、授权和日志记录均在业务请求过程中即时完成；
d) 覆盖要求：本软件单元不覆盖其他业务软件单元，而是被其他业务软件单元调用；
e) 依赖关系：依赖数据库连接、用户表、日志表、系统密钥配置和后端安全认证模块。
4.1.2　 非功能性/性能
　　公共支撑软件单元应满足以下非功能性和性能要求：
a) 有效性要求。
系统应能够准确识别合法用户和非法用户。对于用户名不存在、密码错误、用户停用、令牌无效、令牌过期和权限不足等情况，应返回明确的错误提示，避免用户误认为系统无响应。
b) 效率要求。
用户登录、当前用户识别和权限校验属于高频基础操作，应在较短时间内完成。课程设计阶段，在本地开发环境和普通局域网环境下，单次登录认证和权限校验应满足系统交互的即时性要求，不应造成明显页面卡顿。
c) 安全性要求。
用户密码不得以明文形式保存，应保存哈希后的密码值。用户登录成功后，系统应生成访问令牌，前端后续请求通过令牌识别身份。当前实现中使用 pbkdf2_sha256 方案进行密码哈希，并通过 JWT 生成访问令牌；角色校验函数用于限制不同角色对接口的访问。
d) 完整性要求。
用户记录应包含用户名、邮箱、姓名、密码哈希、角色、启用状态、创建时间和更新时间等基本字段；日志记录应包含用户标识、操作类型、操作对象类型、操作对象标识、操作说明和创建时间等字段，保证认证与审计信息完整。
e) 可靠性要求。
认证失败、权限不足、数据库异常和令牌解析异常时，系统不得崩溃，应返回统一错误信息并中止当前操作。对于关键操作，应尽量记录审计日志，便于后续问题定位。
f) 可维护性要求。
认证、授权、用户管理和日志记录应分别由相对独立的接口、服务和数据模型承担，避免将所有逻辑混写在业务模块中。当前项目后端已经按 api、core、models、schemas、services 分层组织，有利于后续维护。
g) 可扩展性要求。
当前阶段系统主要支持管理员、编辑者、只读用户等基础角色。后续如增加审阅人员、项目管理员、模板管理员等角色，应尽量通过扩展角色判断逻辑实现，不应大幅改动业务接口结构。
4.1.3　 算法
　　本软件单元不涉及复杂数学算法，主要涉及密码哈希算法、密码校验逻辑、JWT 令牌生成逻辑、令牌解析逻辑、角色权限判断逻辑和日志记录逻辑。

4.1.7　 存储分配
　　公共支撑软件单元主要涉及数据库存储、令牌临时存储和运行时内存对象三类存储。用户数据持久化保存在 users 表中，审计日志数据持久化保存在 audit_logs 表中；数据库由 MySQL 提供存储支持，系统通过 SQLAlchemy 数据模型进行访问。
　　访问令牌由后端登录接口生成后返回前端。后端不需要在数据库中保存每一个访问令牌，前端在浏览器端保存令牌，并在后续请求中携带；后端通过密钥解析和校验令牌有效性。
　　运行时内存对象包括用户对象、认证载荷、角色集合和数据库会话对象。上述对象仅在请求处理过程中临时存在，请求结束后由运行环境回收。
4.1.8　 限制条件
　　用户必须通过合法用户名和密码完成登录后，才能访问受保护业务接口。
　　用户访问受保护接口时必须携带有效访问令牌，否则系统应拒绝访问。
　　令牌过期、令牌被篡改或令牌缺少用户标识时，系统应视为未授权访问。
　　被停用用户即使密码正确，也不得登录系统或继续访问受保护接口。
　　自助注册用户的角色范围应受到限制，不允许普通用户自行注册为管理员。
　　管理员不应停用当前正在使用的管理员账号，也不应随意将自身角色修改为非管理员。
　　日志记录应以关键操作为主，课程设计阶段不要求记录每一次普通页面查看行为。
4.1.9　 出错设计
　　公共支撑软件单元应按照错误类型返回明确、可理解的错误提示，并避免将系统密钥、数据库连接、密码哈希等敏感信息暴露给前端。

4.1.10　 尚未解决的问题
　　当前角色权限控制以角色名称判断为主，尚未实现更细粒度的"菜单-按钮-接口-数据范围"权限模型。
　　当前访问令牌主要依赖有效期控制，尚未实现令牌主动注销、令牌黑名单或刷新令牌机制。
　　当前用户注册和登录尚未加入验证码、登录失败次数限制、账号临时锁定等安全增强机制。
　　当前日志审计主要记录关键业务操作，尚未形成完整的日志检索、日志导出和异常告警机制。
　　当前用户管理主要面向课程演示场景，尚未实现复杂组织结构、项目级用户权限继承和多角色绑定。
　　
4.2　 项目管理软件单元 SYSMLDOCGEN-CSU-PROJECT
4.2.1　  功能描述
　　项目管理软件单元的项目唯一标识为 SYSMLDOCGEN-CSU-PROJECT。该软件单元是基于 SysML 模型的文档自动生成系统中用于组织业务数据范围的基础业务单元，主要负责项目的创建、查询、查看、修改、删除以及项目成员管理。模型文件、文档模板、生成文档和后续日志记录均需要依托具体项目进行组织，因此项目管理软件单元为系统后续模型管理、模板管理和文档生成管理提供项目范围约束。
　　本软件单元面向系统管理员、项目创建者以及项目成员使用。管理员可查看和管理系统内全部项目；项目创建者作为项目所有者，可维护自己创建的项目及其成员；普通项目成员根据其在项目中的成员角色获得相应的查看、编辑或管理权限。
　　本软件单元由项目创建子功能、项目查询子功能、项目详情查看子功能、项目更新子功能、项目删除子功能和项目成员管理子功能组成。项目创建子功能负责接收项目名称、项目编号和项目描述，创建项目记录并将当前用户登记为项目所有者；项目查询子功能根据当前用户角色返回其可访问的项目列表；项目详情查看子功能在权限校验通过后返回指定项目的详细信息；项目更新和删除子功能负责维护项目基本信息；项目成员管理子功能负责项目成员列表查询、成员添加、成员角色修改和成员移除。
　　本程序属于 Web 后端业务处理程序，随系统后端服务启动而加载，采用请求驱动方式运行。当用户在前端项目管理页面执行项目创建、项目列表查询、项目详情查看、项目修改、项目删除或成员管理操作时，本程序被调用并同步完成权限校验、数据校验、数据库访问和日志记录。本程序不直接执行模型解析或文档生成，但其输出的项目数据是模型、模板和生成文档等业务数据的基础归属对象。
4.2.2　 非功能性/性能
　　项目管理软件单元应满足以下非功能性和性能要求：
a) 有效性要求。系统应能够准确创建、查询、更新和删除项目记录，并保证项目编号在系统内唯一。对于项目不存在、项目编号重复、成员不存在、成员角色非法和用户权限不足等情况，应给出明确提示。
b) 效率要求。项目列表查询、项目详情查询和成员列表查询属于常用操作，在课程设计阶段的本地开发环境和普通局域网环境下应能够快速返回，不应造成页面长时间无响应。
c) 完整性要求。项目记录应包含项目名称、项目编号、项目描述、项目所有者、创建时间和更新时间等基本信息；项目成员记录应包含项目标识、用户标识、项目成员角色和创建时间等信息。
d) 安全性要求。项目创建、修改、删除和成员管理操作必须在用户认证后进行，并根据系统角色和项目成员角色进行权限判断。未登录用户不得访问项目接口，权限不足用户不得修改或删除项目数据。
e) 可靠性要求。数据库查询、写入或删除操作失败时，系统应返回错误提示并避免产生不一致数据。项目删除时，与项目关联的模型、模板和生成文档等数据应按数据库关系进行级联处理或按系统设计进行统一维护。
f) 可维护性要求。项目管理逻辑应与认证权限、模型管理和文档生成逻辑保持相对独立，通过项目标识与其他业务单元关联，避免将模型解析或文档生成逻辑混入项目管理接口中。
g) 可扩展性要求。当前项目成员角色主要包括 manager、editor、viewer。后续如需增加审阅人员、文档管理员等项目内角色，应通过扩展项目成员角色集合和权限判断逻辑实现。
4.2.3　 算法
　　本软件单元不涉及复杂数学算法，主要涉及项目访问权限判断、项目管理权限判断、项目成员角色校验和项目列表过滤算法。
a) 项目访问权限判断算法。系统首先读取当前用户对象和目标项目对象。若当前用户为系统管理员，则允许访问；若当前用户为项目所有者，则允许访问；否则查询 project_members 表中是否存在当前用户与该项目的成员关系。若存在成员关系，则允许访问；否则拒绝访问。
b) 项目管理权限判断算法。系统首先查询目标项目是否存在。若项目不存在，则返回项目不存在错误；若当前用户为系统管理员或项目所有者，则允许执行管理操作；否则查询当前用户在该项目中的成员角色。仅当成员角色为 manager 时，允许执行成员管理或项目维护操作，否则返回权限不足。
c) 项目成员角色校验算法。系统接收新增或修改项目成员请求后，判断成员角色是否属于 manager、editor、viewer 三类允许值。若角色不属于允许集合，则拒绝操作并返回角色非法提示。
d) 项目列表过滤算法。系统查询项目列表时，若当前用户为管理员，则返回全部项目；若当前用户不是管理员，则只返回当前用户作为项目所有者的项目，或当前用户已加入 project_members 成员表的项目。

4.2.7　 存储分配
　　项目管理软件单元主要涉及数据库存储、运行时对象和日志记录三类存储分配。
a) 数据库存储。项目基本信息持久化保存在 projects 表中，项目成员关系持久化保存在 project_members 表中。projects 表保存项目名称、项目编号、项目描述、所有者和时间信息；project_members 表保存项目与用户之间的成员关系以及项目内角色。
b) 关联数据存储。项目作为模型、模板和生成文档等业务数据的上层归属对象，与模型数据、模板数据和生成文档记录存在关联关系。项目删除时，应依据数据库关系和系统设计要求同步处理关联数据。
c) 运行时对象。后端处理项目请求时会在内存中临时保存当前用户对象、项目对象、项目成员对象、请求数据结构和数据库会话对象。请求处理完成后，这些临时对象由运行环境回收。
d) 日志存储。项目创建、更新、删除和成员管理等关键操作应写入审计日志表，记录操作用户、操作类型、操作对象和说明信息。
4.2.8　 限制条件
　　项目管理软件单元运行中受到以下限制：
　　用户必须登录后才能访问项目管理相关接口。
　　只有管理员和 author 角色用户可以创建项目。
　　项目编号 code 必须保持系统内唯一，不允许创建重复项目编号。
　　非管理员用户只能查看自己创建的项目或自己作为成员参与的项目。
　　只有管理员、项目所有者或项目内 manager 角色用户可以执行项目更新和成员管理操作。
　　只有管理员或项目所有者可以删除项目。
　　项目成员角色只能为 manager、editor 或 viewer。
　　项目所有者不需要作为普通成员重复添加到项目成员表中。
　　同一项目中同一用户只能存在一条成员关系记录。
　　当前课程设计阶段以项目基本信息和成员管理为主，不实现复杂项目流程审批、项目归档、项目版本管理等扩展功能

4.2.10　 尚未解决的问题
　　当前项目管理软件单元已经能够满足课程设计阶段的项目创建、项目查询、项目维护和成员管理要求，但仍存在以下后续可完善内容：
a) 当前项目管理主要维护项目基本信息，尚未实现项目状态流转、项目归档、项目阶段管理等扩展功能。
b) 当前项目内角色主要为 manager、editor、viewer，尚未实现更细粒度的数据范围权限和按钮级权限。
c) 当前项目成员管理以单个成员添加和修改为主，尚未实现批量导入成员、批量授权和成员变更历史追踪。
d) 当前项目删除操作依赖数据库关联关系和后端逻辑处理，后续如用于真实工程场景，应增加删除前确认、软删除、归档和恢复机制。
e) 当前项目查询以列表返回为主，尚未实现复杂搜索、分页筛选和统计分析功能。
f) 当前项目管理功能可满足课程设计验收，但如后续扩展为实际工程系统，还需进一步完善项目生命周期管理、成员权限继承和项目数据审计机制。
　
4.3　 模型管理与解析软件单元 SYSMLDOCGEN-CSU-MODEL
4.3.1　  功能描述
　　模型管理与解析软件单元的项目唯一标识为 SYSMLDOCGEN-CSU-MODEL。该软件单元是基于 SysML 模型的文档自动生成系统的核心业务单元，主要负责模型文件上传、模型元数据维护、模型版本管理、模型文件解析、模型元素与关系结构化存储以及模型版本差异比较。该单元直接决定系统能否将外部 SysML、UML、XMI、XML 或 JSON 模型文件转换为系统内部可复用的数据结构，是后续视图浏览、轻量编辑、模板填充和文档自动生成的基础。
　　本软件单元面向建模人员、项目管理员和具有模型编辑权限的项目成员使用。用户在项目范围内上传模型文件，系统首先检查项目是否存在以及当前用户是否具有对应项目的管理或编辑权限；校验通过后，系统保存原始模型文件，调用模型解析服务抽取模型元素和模型关系，并将解析结果写入数据库。若同一项目下上传同名模型，系统根据已有模型版本自动生成新的版本号，从而支持后续版本追踪和差异比较。
　　本软件单元由模型上传子功能、模型列表查询子功能、模型解析子功能、模型版本管理子功能、模型版本比较子功能和模型访问权限判断子功能组成。模型上传子功能负责接收项目编号、模型名称、模型描述和上传文件；模型解析子功能负责根据文件后缀选择 JSON 或 XML 解析方式；模型版本管理子功能负责根据同名模型历史记录生成版本号；模型版本比较子功能负责比较两个模型版本中的元素和关系差异；权限判断子功能负责保证用户只能访问其所属项目范围内的模型数据。
　　本程序属于 Web 后端核心业务处理程序，随系统后端服务启动而加载，采用请求驱动方式运行。当用户在前端模型管理页面执行模型上传、模型列表查询、模型版本比较等操作时，本程序被调用并同步完成项目权限校验、文件保存、模型解析、数据库写入和日志记录。本程序不直接负责文档模板填充和文档文件导出，但其输出的模型元素与关系数据是文档生成软件单元的主要输入来源。
4.3.2　 非功能性/性能
　　模型管理与解析软件单元应满足以下非功能性和性能要求：
a) 有效性要求。系统应能够正确接收并保存用户上传的模型文件，支持对 SysML、UML、XMI、XML、JSON 等常见模型文件进行解析。对于项目不存在、权限不足、文件格式异常或解析不到模型元素等情况，应返回明确提示。
b) 效率要求。模型上传与解析操作属于相对耗时操作，但在课程设计阶段的示例模型规模下，应能在用户可接受时间内完成。模型列表查询、版本比较和模型访问权限判断应尽量减少不必要的重复查询。
c) 完整性要求。模型记录应包含所属项目、模型名称、模型描述、原始文件名、服务器存储路径、版本号、上传用户、解析状态和创建时间等信息；模型元素记录应包含元素唯一标识、元素名称、元素类型、说明信息、父元素标识和原始 JSON 信息；模型关系记录应包含源元素、目标元素、关系类型和关系标签等信息。
d) 安全性要求。模型上传和模型元素维护必须在用户登录后进行，且需要结合系统角色和项目内角色进行权限控制。用户不得上传或访问无权限项目下的模型数据。上传文件应保存到后端指定目录，不允许前端直接操作服务器文件系统。
e) 可靠性要求。模型解析失败或未解析到模型元素时，系统应终止本次上传操作并返回错误提示，不应写入不完整的模型解析数据。数据库写入模型记录、元素记录和关系记录时，应保证数据之间的对应关系。
f) 可维护性要求。模型上传、文件解析、结构化存储和版本比较逻辑应相对独立。解析规则集中在模型解析服务中维护，接口层主要负责请求接收、权限校验和服务调用，避免将不同文件格式解析规则分散在多个接口中。
g) 可扩展性要求。当前阶段重点支持 JSON、XML、XMI、UML 和 SysML 文件的基础解析。后续如需支持更多建模工具导出的专有格式，应优先在模型解析服务中扩展解析策略，不应大幅改动模型管理接口结构。
4.3.3　 算法
　　本软件单元涉及模型上传与版本生成算法、模型文件类型选择算法、JSON 模型解析算法、XML 模型解析算法、包含关系补充算法、关系去重算法和模型版本比较算法。
a) 模型上传与版本生成算法。系统接收上传请求后，首先查询项目是否存在，并判断当前用户是否具有项目 manager 或 editor 权限。权限通过后，系统根据原始文件后缀生成唯一存储文件名，将文件保存到上传目录。随后系统查询同一项目下同名模型的最新版本，若不存在历史版本，则版本号设置为 1；若存在历史版本，则新版本号为最新版本号加 1。
b) 模型文件类型选择算法。系统读取上传文件后缀并转换为小写。若后缀为 .json 或 .mms，则调用 JSON 解析逻辑；若后缀为 .xml、.xmi、.uml 或 .sysml，则调用 XML 解析逻辑；若后缀不在上述集合中，则先尝试 JSON 解析，失败后再尝试 XML 解析。
c) JSON 模型解析算法。系统将 JSON 文本转换为字典或列表对象，并递归遍历其中的字典结构。对于包含 id、_id、elementId、xmi:id 等标识字段的对象，系统将其作为候选模型元素。随后提取元素标识、元素名称、元素类型、说明文档和父元素标识，并根据 source、target、clientId、supplierId、from、to 等字段抽取模型关系。
d) XML 模型解析算法。系统使用 XML 解析器读取模型文件根节点，并递归访问所有子节点。对于带有 id 或 xmi:id 的节点，系统将其转换为模型元素，并根据节点属性提取元素类型、名称、说明和父元素标识。对于 supplier、client、type、href 或以 Id 结尾的属性，系统尝试拆分引用值并生成模型关系。
e) 包含关系补充算法。系统遍历已解析的模型元素。若某个元素存在 parent_uid，且该父元素标识在当前模型元素集合中存在，则系统补充一条从父元素到子元素的 containment 关系，关系标签为 contains。
f) 关系去重算法。系统以 source_uid、target_uid 和 relation_type 组成关系唯一键。遍历解析得到的关系列表时，若关系源、关系目标均存在且唯一键未出现过，则保留该关系；若唯一键已存在，则丢弃重复关系。
g) 模型版本比较算法。系统分别读取基准模型和目标模型的元素集合，并以 element_uid 建立映射。目标模型中存在而基准模型中不存在的元素记为新增元素；基准模型中存在而目标模型中不存在的元素记为删除元素；两个模型均存在但名称、类型、说明或父元素标识发生变化的元素记为变更元素。关系比较时，以源元素、目标元素、关系类型和标签组成关系集合，并计算新增关系和删除关系。

4.3.7　 存储分配
　　模型管理与解析软件单元主要涉及数据库存储、文件目录存储和运行时中间对象三类存储分配。
a) 数据库存储。模型元数据持久化保存在 sysml_models 表中，模型元素数据持久化保存在 model_elements 表中，模型关系数据持久化保存在 model_relations 表中。三类数据通过 model_id 和 project_id 建立关联，保证模型文件、解析结果和项目之间具有明确对应关系。
b) 文件目录存储。用户上传的原始模型文件保存在后端配置的上传目录中。数据库中的 stored_path 字段保存服务器端文件路径，source_filename 字段保存用户上传时的原始文件名。
c) 运行时中间对象。模型解析过程中，系统会在内存中临时构造 ParsedModel、ParsedElement 和 ParsedRelation 对象。解析完成后，这些中间对象被转换为数据库记录；请求处理完成后，中间对象由运行环境回收。
d) 日志存储。模型上传与解析成功后，系统应写入审计日志，记录上传用户、模型标识和解析元素数量等信息，便于后续审计和问题定位。
4.3.8　 限制条件
　　模型管理与解析软件单元运行中受到以下限制：
　　用户必须登录后才能访问模型管理相关接口。
　　只有 admin 或 author 系统角色用户可以发起模型上传操作。
　　用户必须在所属项目中具有 manager 或 editor 项目角色，才能向该项目上传模型文件。
　　上传模型必须归属于一个已存在项目，项目不存在时不得保存模型记录。
　　模型解析必须至少得到一个模型元素，否则系统应拒绝本次上传并提示检查文件格式。
　　模型版本比较仅允许在同一项目下的两个模型版本之间进行，不允许跨项目比较。
　　模型元素名称、元素类型和关系类型写入数据库时应进行长度限制，避免超过数据库字段长度。
　　当前解析逻辑重点抽取常见模型元素、元素属性和引用关系，不保证完整覆盖所有专业建模工具的全部 SysML 语义。
　　当前课程设计阶段以中小规模模型文件为主要演示对象，不针对超大规模模型文件、高并发上传和分布式解析进行复杂优化。

4.3.10　 尚未解决的问题
　　当前模型管理与解析软件单元已经能够满足课程设计阶段的模型上传、基础解析、版本管理和差异比较要求，但仍存在以下后续可完善内容：
a) 当前模型解析以通用 JSON/XML/XMI 字段抽取为主，尚未针对特定建模工具的完整 SysML 语义进行深度适配。
b) 当前模型版本管理采用同一项目下同名模型版本号递增方式，尚未实现模型版本回滚、版本锁定和版本分支管理。
c) 当前模型比较主要比较元素字段和关系集合差异，尚未实现图形化差异展示、语义级差异分析和复杂模型变更影响分析。
d) 当前上传模型文件直接保存到服务器目录，尚未实现文件大小分级控制、病毒扫描、文件完整性校验和上传过程断点续传。
e) 当前解析过程为同步处理，适合课程演示和中小型模型文件；对于大型模型文件，后续可扩展为异步解析任务并增加解析进度反馈。
f) 当前模型数据可满足后续文档生成需要，但如后续扩展为实际工程系统，还需进一步完善模型语义映射、模型一致性校验和模型数据质量评估机制。
　　
4.4　 视图浏览与轻量编辑软件单元 SYSMLDOCGEN-CSU-VIEW
4.4.1　  功能描述
　　视图浏览与轻量编辑软件单元的项目唯一标识为 SYSMLDOCGEN-CSU-VIEW。该软件单元用于在 Web 端对已解析的 SysML 模型数据进行展示、查询和轻量维护，是模型管理与文档生成之间的重要衔接单元。模型管理与解析软件单元完成模型文件上传、解析和入库后，视图浏览与轻量编辑软件单元负责将模型元素、模型关系和元素说明以列表、详情和关系图等形式呈现给用户，使用户能够在不依赖专业建模工具的情况下核对模型内容。
　　本软件单元不以完整替代 SysML 专业建模工具为目标，而是重点支持模型内容的浏览、检索、关系查看和少量说明性字段维护。对于具备相应项目权限的用户，系统允许其对模型元素名称和文档说明字段进行编辑；对于只读用户，系统只提供查看能力，不允许保存修改。
4.4.2　 非功能性/性能
　　说视图浏览与轻量编辑软件单元应满足以下非功能性和性能要求：
a) 有效性要求。系统应能够正确展示指定模型版本下的模型元素和模型关系，返回的数据应与数据库中解析保存的模型结构一致。用户选择模型后，应能够看到元素名称、元素类型、说明、父元素标识和关系信息。
b) 效率要求。模型元素列表和关系图加载应满足前端页面交互的即时性要求。课程设计阶段，对于中小规模示例模型，元素列表和关系图数据应在普通本地开发环境或局域网环境下较快返回，不应造成明显页面卡顿。
c) 安全性要求。所有模型视图访问和元素编辑操作均必须经过登录认证和项目权限校验。未登录用户不得访问模型视图数据；无项目访问权限的用户不得查看模型元素；无编辑权限的用户不得保存模型元素修改。
d) 完整性要求。模型元素编辑时应保留元素唯一标识、元素类型、父元素标识和原始 JSON 数据等系统字段，不应因轻量编辑破坏模型解析结果的基本结构。
e) 可靠性要求。模型不存在、元素不存在、权限不足或数据库更新失败时，系统应返回明确错误信息，并避免前端显示不完整或错误的视图数据。
f) 可用性要求。前端页面应以列表、详情和关系视图等方式展示模型内容，使用户能够快速定位元素、查看说明并进行必要修改。保存成功或失败时应给出明确反馈。
g) 可维护性要求。视图浏览、关系图构造、元素编辑和权限校验应保持职责分离，便于后续扩展模型过滤、元素搜索、图形布局优化和更复杂的模型编辑能力。
4.4.3　 算法
　　本软件单元不涉及复杂数学算法，主要涉及模型元素读取算法、模型关系图构造算法、模型元素轻量编辑算法和模型访问权限校验算法。
a) 模型元素读取算法。系统接收模型编号后，首先调用模型访问权限校验逻辑判断当前用户是否能够访问该模型；校验通过后，从 model_elements 表中读取 model_id 对应的元素记录，并按照元素类型和元素名称进行排序；后端将查询结果转换为 ModelElementOut 数据结构后返回前端。
b) 模型关系图构造算法。系统接收模型编号后，先校验用户是否具有模型访问权限；校验通过后，分别查询 model_elements 表和 model_relations 表，获取该模型版本的全部元素和关系；随后将元素列表转换为图节点集合，将关系列表转换为图边集合，最终封装为 ModelGraphOut 数据结构返回前端。
c) 模型元素轻量编辑算法。用户提交元素名称或说明字段修改请求后，系统根据 element_id 查询模型元素；若元素不存在，则返回错误；若元素存在，则进一步校验当前用户对该元素所属模型和项目是否具有编辑权限；权限通过后，系统只更新允许修改的 name 和 documentation 字段，并提交数据库事务。保存完成后，系统刷新元素对象并记录操作日志。
d) 模型访问权限校验算法。系统根据 model_id 查询对应模型及所属项目；若模型不存在，则返回模型不存在；若模型存在，则调用项目成员权限判断逻辑，确认当前用户是否为管理员、项目负责人或项目成员；若用户不具备访问权限，则拒绝访问；若权限满足要求，则返回模型对象供后续查询或编辑操作使用。

4.4.7　 存储分配
　　视图浏览与轻量编辑软件单元主要涉及数据库存储和运行时内存对象两类存储分配。
a) 数据库存储。模型元素列表和关系图数据来源于 model_elements 表和 model_relations 表。元素轻量编辑操作主要更新 model_elements 表中的 name 和 documentation 字段。编辑操作完成后，系统通过审计日志机制在 audit_logs 表中记录操作信息。
b) 运行时内存对象。系统在处理元素列表查询或关系图查询时，会在内存中临时构造 ModelElementOut、ModelRelationOut 和 ModelGraphOut 对象；在处理元素更新时，会在内存中临时保存 ModelElementUpdate 请求对象、当前用户对象和数据库会话对象。请求结束后，相关临时对象由运行环境回收。
c) 文件存储。该软件单元不直接读写模型源文件，也不直接写入生成文档文件。模型源文件的保存由模型管理与解析软件单元负责，生成文档文件的保存由文档生成与导出软件单元负责。
4.4.8　 限制条件
　　视图浏览与轻量编辑软件单元运行中受到以下限制：
　　用户必须登录后才能查看模型元素列表、模型关系图和元素详情；
　　用户只能查看其有权访问的项目范围内的模型数据；
　　模型关系图展示依赖模型解析结果，若模型尚未成功解析或解析结果为空，则无法展示有效关系图；
　　轻量编辑仅允许修改模型元素名称和说明字段，不允许通过本单元修改元素唯一标识、元素类型、父元素标识和元素关系；
　　元素编辑操作要求用户具有系统角色 admin 或 author，并且在项目内具有 manager 或 editor 权限；
　　本软件单元不承担完整 SysML 建模能力，不支持复杂图形建模、拖拽建模、关系新增删除和模型语义校验；
　　当前阶段关系图展示以结构化数据返回为主，图形布局和交互效果由前端页面实现，后端不负责复杂图布局计算。
4.4.9　 出错设计
　　视图浏览与轻量编辑软件单元应按照错误类型返回明确提示，并避免在异常情况下破坏模型元素和关系数据。

4.4.10　 尚未解决的问题
　　当前视图浏览与轻量编辑软件单元已经能够满足课程设计阶段的模型元素展示、关系图数据返回和元素说明维护要求，但仍存在以下后续可完善内容：
a) 当前轻量编辑仅支持元素名称和说明字段维护，尚未支持模型关系的新增、删除和修改；
b) 当前关系图数据由后端返回元素和关系列表，复杂图形布局、节点折叠、分层展示和大规模图性能优化仍需在后续前端实现中进一步完善；
c) 当前元素列表主要按类型和名称排序，尚未实现多条件过滤、全文搜索和按关系路径定位元素等高级查询能力；
d) 当前编辑操作直接更新数据库中的元素说明，尚未实现元素级修改历史、差异对比和回滚功能；
e) 当前系统不对用户修改后的说明文本进行复杂语义校验，后续如面向实际工程项目，可增加字段规范、敏感词过滤和模型语义一致性检查；
f) 当前关系图主要服务课程演示和文档生成前的数据核对，尚未形成完整的在线 SysML 建模器能力。

4.5　 模板管理软件单元 SYSMLDOCGEN-CSU-TEMPLATE
4.5.1　  功能描述
　　模板管理软件单元的项目唯一标识为 SYSMLDOCGEN-CSU-TEMPLATE。该软件单元用于对文档生成过程中所需的模板进行统一管理，是连接模型数据与文档生成结果的重要配置单元。用户可以通过该软件单元创建全局模板或项目模板，维护模板名称、模板说明和模板内容，并在后续文档生成过程中选择相应模板作为文档内容组织规则。
　　本软件单元不直接完成模型文件解析，也不直接负责最终文档导出；其主要职责是维护文档模板的主记录、历史版本和预览结果，为文档生成管理软件单元提供可复用、可追踪的模板数据。模板内容采用文本形式保存，可包含 HTML 结构和模板占位表达式，文档生成时由后端根据模型数据进行填充。
4.5.2　 非功能性/性能
　　模板管理软件单元应满足以下非功能性和性能要求：
a) 有效性要求。系统应能够正确保存模板名称、说明、内容、所属项目和版本号，并能够根据项目范围返回全局模板和项目模板。模板修改后，模板主记录应与最新版本保持一致。
b) 效率要求。模板列表查询、模板版本查询和模板预览属于常用页面操作，在课程设计阶段的本地开发环境和普通局域网环境下应能够快速返回。模板内容一般为文本数据，不应造成页面长时间无响应。
c) 完整性要求。模板主记录与模板历史版本记录之间应保持对应关系。每次创建模板、修改模板或回滚模板后，应保存相应版本记录，以保证模板历史可追踪。
d) 安全性要求。模板创建、修改、删除和回滚操作必须经过用户认证、系统角色校验和项目编辑权限校验。无权限用户不得修改项目模板或删除模板。
e) 可靠性要求。模板不存在、项目不存在、模板版本不存在、权限不足、模板已被生成文档引用或数据库提交失败时，系统应返回明确错误提示，并避免模板数据处于不一致状态。
f) 可用性要求。系统应提供模板预览和版本查看能力，使用户能够在生成文档前检查模板内容，并能够在模板配置错误时回滚到历史版本。
g) 可维护性要求。模板主记录、模板版本记录、模板预览和权限校验逻辑应保持相对独立，便于后续扩展模板分类、模板复制、模板导入导出和复杂模板语法校验能力。
4.5.3　 算法
　　本软件单元不涉及复杂数学算法，主要涉及模板版本保存算法、模板更新版本号算法、模板回滚算法、模板预览渲染算法和模板访问权限校验算法。
a) 模板版本保存算法。系统在创建模板、修改模板或回滚模板后，根据模板主记录中的模板编号和版本号检查当前版本是否已经存在。如果同一模板编号和版本号的版本记录不存在，则创建 DocumentTemplateVersion 记录，将模板名称、说明、内容、创建人和创建时间写入版本表；若版本记录已经存在，则不重复创建。
b) 模板更新版本号算法。用户提交模板更新请求后，系统比较请求中的模板名称、说明和内容与当前模板主记录是否一致。若名称、说明或内容任一字段发生变化，则将模板版本号加 1，并保存新的历史版本；若未发生变化，则不增加版本号。
c) 模板回滚算法。用户选择模板历史版本后，系统根据 template_id 和 version_id 查询对应版本记录。若历史版本存在且用户具有写权限，则将历史版本中的名称、说明和内容写回模板主记录，同时将当前模板版本号加 1，并将回滚后的内容作为一个新的版本保存。
d) 模板预览渲染算法。系统接收前端提交的模板标题和模板内容后，构造示例模型数据、示例元素数据、示例关系数据和统计数据，然后使用模板渲染机制将示例数据填充到模板内容中，生成 HTML 预览结果并返回前端。该流程不写入模板主表和模板版本表。
e) 模板访问权限校验算法。对于全局模板，系统允许具备系统角色权限的用户进行相应操作；对于项目模板，系统首先查询项目是否存在，再判断当前用户是否具有该项目的 manager 或 editor 权限。若权限不足，则拒绝创建、修改、删除或回滚操作。

4.5.7　 存储分配
　　模板管理软件单元主要涉及数据库存储、前端运行时状态和日志存储三类资源。
a) 数据库存储。模板主记录存储在 document_templates 表中，模板历史版本存储在 document_template_versions 表中。模板删除时还需检查 generated_documents 表中是否存在对该模板的引用。
b) 前端运行时状态。前端模板管理页面临时保存模板列表、当前选中项目、模板编辑表单、历史版本列表和模板预览 HTML。该部分数据仅用于页面展示和交互，刷新页面后需要重新从后端加载。
c) 日志存储。模板创建、修改、删除、回滚和默认模板创建等关键操作写入 audit_logs 表，记录操作用户、操作动作、目标对象和说明信息。
d) 文件存储。模板管理软件单元本身不直接保存上传文件或生成文档文件。模板内容以文本形式存储于数据库中，文档文件生成由文档生成与导出软件单元负责。
4.5.8　 限制条件
　　模板管理软件单元运行中受到以下限制：
　　用户必须登录后才能查询模板、预览模板或查看模板版本；
　　模板创建、修改、删除和回滚操作仅允许具备 admin 或 author 角色的用户执行；
　　对于项目模板，用户还必须具有该项目的 manager 或 editor 权限，否则不得修改项目模板；
　　模板名称和模板内容不能为空，模板内容应符合后端模板渲染的基本语法要求；
　　已被生成文档引用的模板不得删除，以保证历史生成文档能够追溯其模板来源；
　　模板回滚不是简单地降低版本号，而是将历史版本内容恢复为当前内容并生成新的版本记录；
　　模板预览仅使用示例模型数据，不能完全代表真实模型生成效果；真实生成效果仍以文档生成软件单元使用具体模型数据后的结果为准；
　　当前课程设计阶段主要支持文本或 HTML 模板管理，尚未实现复杂可视化模板编辑器、模板导入导出和模板语法静态检查。

4.5.10　 尚未解决的问题
　　当前模板管理软件单元已经能够满足课程设计阶段的模板创建、查询、修改、删除、预览、版本管理和回滚要求，但仍存在以下后续可完善内容：
a) 当前模板编辑以文本编辑为主，尚未实现图形化或所见即所得模板编辑器；
b) 当前模板预览使用示例模型数据，尚未支持用户选择真实模型进行预览；
c) 当前模板语法主要依赖运行时渲染发现错误，尚未实现模板语法静态检查和错误定位提示；
d) 当前模板版本管理支持查看和回滚，但尚未实现两个版本之间的差异比较；
e) 当前模板删除只检查生成文档引用，尚未提供模板归档、停用或软删除机制；
f) 当前模板权限主要依赖项目角色，尚未细化到单个模板级别的共享、只读、编辑和发布审批权限；
g) 当前模板内容存储在数据库中，尚未实现模板文件导入、导出和跨项目复制功能。
　　
4.6　 文档生成与导出软件单元 SYSMLDOCGEN-CSU-DOCGEN
4.6.1　  功能描述
　　文档生成与导出软件单元的项目唯一标识为 SYSMLDOCGEN-CSU-DOCGEN。该软件单元是基于 SysML 模型的文档自动生成系统中负责将模型数据、模板内容和用户输入标题组合为工程文档的核心业务单元，主要完成文档生成、生成文档查询、生成文档详情查看、生成文档删除以及 HTML、DOCX、PDF 格式导出等功能。
　　本软件单元面向系统管理员、文档编写人员和具有项目编辑权限的项目成员使用。用户在文档生成页面选择项目、模型版本和文档模板，并输入文档标题后，系统首先校验项目、模型和模板是否存在，再校验当前用户是否具有该项目的管理或编辑权限；校验通过后，系统读取模型元素和模型关系，构造模板渲染上下文，将模型数据填充至模板中生成 HTML 内容，并保存生成文档记录。
　　本软件单元由文档生成子功能、生成文档列表查询子功能、生成文档详情查看子功能、文档导出子功能、生成文档删除子功能和文档渲染子功能组成。文档生成子功能负责根据项目、模型和模板创建生成文档记录；列表查询子功能负责返回当前用户可访问的生成文档；详情查看子功能负责返回指定生成文档内容；导出子功能负责根据用户选择的格式返回 HTML、DOCX 或 PDF 文件；删除子功能负责删除生成文档记录并清理对应文件；文档渲染子功能负责将模型数据组织为模板可使用的上下文变量。
　　本程序属于 Web 后端核心业务处理程序，随系统后端服务启动而加载，采用请求驱动方式运行。当用户在前端文档页面执行文档生成、文档列表查看、文档详情查看、文档导出或文档删除操作时，本程序被调用并同步完成权限校验、数据查询、模板渲染、文件写入、数据库记录维护和日志记录。本程序依赖项目管理、模型管理、模板管理、文件存储和公共支撑软件单元，是实现"模型驱动文档自动生成"的关键环节。
4.6.2　 非功能性/性能
　　文档生成与导出软件单元应满足以下非功能性和性能要求：
a) 有效性要求。系统应能够根据用户选择的项目、模型和模板生成文档内容，并保证生成文档与所选模型版本、模板版本和项目范围一致。对于项目不存在、模型不存在、模板不存在、权限不足或导出格式不支持等情况，应返回明确提示。
b) 效率要求。文档生成和导出属于用户直接触发的业务操作，在课程设计阶段的本地开发环境和普通局域网环境下，应在可接受时间内完成响应。对于元素数量较多的模型，DOCX 和 PDF 导出应采取行数限制或摘要导出方式，避免长时间阻塞。
c) 完整性要求。生成文档记录应保存项目标识、模型标识、模板标识、标题、生成状态、HTML 内容、文件路径、创建用户和创建时间等信息，保证文档来源和生成过程可追溯。
d) 安全性要求。文档生成、删除和导出操作必须在用户认证后进行，并根据项目角色进行权限判断。未登录用户不得访问生成文档；非项目成员不得查看或导出项目文档；无编辑权限用户不得生成或删除文档。
e) 互操作性要求。系统应支持 HTML、DOCX 和 PDF 三类常用文档输出格式，便于用户在线预览、下载归档和课程验收展示。
f) 可靠性要求。模板渲染、文件写入、数据库提交或文件下载失败时，系统应返回错误提示并避免产生不可追踪的中间状态。生成文档删除时，应先保证数据库记录正确处理，再尽量清理服务器端文件。
g) 可维护性要求。文档生成逻辑、导出逻辑和数据访问逻辑应保持相对独立。模板渲染、DOCX 导出和 PDF 导出应分别由独立函数承担，便于后续扩展更多文档格式。
4.6.3　 算法
　　本软件单元不涉及复杂数学算法，主要涉及模板渲染算法、文档生成算法、生成文档访问过滤算法、文档导出格式选择算法和安全文件名生成算法。
a) 模板渲染算法。系统根据模型编号查询模型元素和模型关系，统计元素总数、关系总数和主要元素类型，构造包含 title、model、elements、relations、element_names 和 stats 的上下文对象，再使用模板引擎将上下文数据填充到模板内容中，生成 HTML 文档内容。
b) 文档生成算法。系统接收项目编号、模型编号、模板编号和标题，依次校验项目、模型和模板是否存在，校验当前用户是否具有项目 manager 或 editor 权限。校验通过后调用模板渲染算法生成 HTML 内容，将 HTML 写入生成文件目录，并创建 GeneratedDocument 数据库记录。
c) 生成文档访问过滤算法。系统查询生成文档列表时，若当前用户为管理员，则可返回全部生成文档；若当前用户不是管理员，则仅返回其拥有项目或参与项目中的生成文档。查询单个文档时，也需要校验当前用户是否具有该文档所属项目的访问权限。
d) 文档导出格式选择算法。系统接收导出格式参数 fmt，并转换为小写。若 fmt 为 html，则返回 HTML 文件；若 fmt 为 docx，则调用 DOCX 摘要导出函数生成 Word 文件；若 fmt 为 pdf，则调用 PDF 摘要导出函数生成 PDF 文件；若 fmt 不属于上述三类，则返回格式不支持错误。
e) 大模型导出摘要算法。对于 DOCX 和 PDF 导出，系统设置最大导出行数限制。DOCX 导出默认展示前 300 条元素和关系，PDF 导出默认展示前 120 条元素，用于降低大模型导出时的性能压力。
f) 安全文件名生成算法。系统根据生成文档标题构造下载文件名，对非字母数字字符进行替换，并限制文件名长度，避免特殊字符影响服务器文件保存和浏览器下载。

4.6.7　 存储分配
　　文档生成与导出软件单元主要涉及数据库存储、文件系统存储、运行时对象和日志记录四类存储分配。
a) 数据库存储。生成文档元数据和 HTML 内容持久化保存在 generated_documents 表中。该表保存项目编号、模型编号、模板编号、标题、状态、HTML 内容、文件路径、创建用户和创建时间等信息。
b) 文件系统存储。生成文档的 HTML 文件保存在后端配置的 generated_path 目录下。用户导出 DOCX 或 PDF 时，系统在该目录下临时或持久生成对应格式文件，并通过文件响应返回给前端。
c) 运行时对象。文档生成过程中会在内存中临时保存项目对象、模型对象、模板对象、模型元素列表、模型关系列表、模板上下文和 HTML 字符串。请求处理完成后，上述临时对象由运行环境回收。
d) 日志存储。文档生成和文档删除等关键操作应写入审计日志表，记录操作用户、操作类型、操作对象、文档标题和操作时间等信息。
4.6.8　 限制条件
　　用户必须登录后才能访问生成文档相关接口。
　　只有管理员和 author 角色用户可以发起文档生成和文档删除操作。
　　生成文档时，项目、模型和模板必须同时存在，且模型必须属于所选项目。
　　生成文档时，当前用户必须具有项目 manager 或 editor 权限。
　　非项目成员不得查看、导出或删除该项目下的生成文档。
　　文档导出格式仅支持 html、docx 和 pdf，其他格式不予处理。
　　DOCX 和 PDF 导出采用摘要导出方式，存在最大导出行数限制；完整内容应优先使用 HTML 导出。
　　文件路径由后端生成和维护，用户不得直接提交服务器端文件路径。
　　当前课程设计阶段以文档生成、预览和导出为主，不实现多人在线协同编辑、复杂排版控制和文档全文版本比对。
4.6.9　 出错设计
　　文档生成与导出软件单元应按照错误类型返回明确提示，并避免因单个文档生成或导出失败导致系统整体中断。

4.6.10　 尚未解决的问题
　　当前文档生成以 HTML 渲染和摘要导出为主，尚未实现复杂 Word 样式、分页、页眉页脚、目录和图表自动编号等高级排版能力。
　　当前 DOCX 和 PDF 导出采用摘要策略，模型规模较大时不能完整展示全部元素和关系，后续可增加分页导出或异步导出机制。
　　当前生成文档记录主要保存 HTML 内容和文件路径，尚未实现生成文档版本管理、历史比对和回滚。
　　当前文档生成过程为同步处理，尚未引入任务队列、进度条和生成失败重试机制。
　　当前模板渲染变量相对固定，后续如需支持更多 SysML 视图、图形化关系图或需求追踪矩阵，需要扩展模板上下文数据结构。
　　当前文档导出文件保存在服务器目录中，尚未实现文件定期清理、存储容量监控和下载权限长期控制。
　　当前文档生成与导出功能可满足课程设计验收，但如后续扩展为实际工程系统，还需进一步完善排版能力、异步任务处理、文档版本追踪和文件存储安全。

4.7　 日志审计软件单元 SYSMLDOCGEN-CSU-AUDIT
4.7.1　  功能描述
　　日志审计软件单元的项目唯一标识为 SYSMLDOCGEN-CSU-AUDIT。该软件单元是基于 SysML 模型的文档自动生成系统中用于记录、保存和查询系统关键操作信息的支撑性业务单元，主要负责对用户登录、项目维护、模型上传与解析、模板维护、文档生成与导出等操作形成审计记录，并向管理员提供日志查询能力。
　　本软件单元面向系统管理员和系统内部业务单元使用。系统管理员可通过日志管理页面查看系统运行过程中产生的操作记录，辅助完成问题定位、运行审计和责任追踪；其他业务软件单元在完成关键业务操作后调用日志写入服务，将操作用户、操作动作、操作对象类型、操作对象标识和操作说明写入审计日志表。
4.7.2　 非功能性/性能
　　日志审计软件单元应满足以下非功能性和性能要求：
a) 有效性要求。系统应能够准确记录关键业务操作，并保证日志记录中包含操作用户、操作动作、操作对象类型、操作对象标识、操作说明和创建时间等必要信息。管理员查询日志时，系统应按照指定条件返回匹配的日志记录。
b) 效率要求。日志写入属于业务操作的附加处理，应尽量减少对主业务流程的影响；日志查询应支持限制返回数量，避免一次性返回过多记录造成页面卡顿。课程设计阶段单次查询默认返回有限数量日志，满足教学演示和问题定位需要。
c) 完整性要求。审计日志表应保存能够描述操作行为的核心字段。对于无明确操作对象的日志，target_type、target_id 和 message 字段可为空，但 action 和 created_at 字段应保持有效。
d) 安全性要求。日志查询接口应仅允许管理员访问，普通作者和只读用户不得查看全局操作日志。日志内容不得泄露用户密码、令牌、数据库连接信息等敏感内容。
e) 可靠性要求。日志查询失败时应返回明确错误提示；日志写入失败时应避免造成系统整体崩溃。对于数据库提交异常，应由调用方或数据库会话管理机制进行回滚处理。
f) 可维护性要求。日志写入逻辑应封装为独立服务函数，避免在各业务接口中重复编写日志对象构造和提交逻辑。日志查询逻辑应集中在日志管理接口中，便于后续扩展更多筛选条件。
g) 可扩展性要求。当前系统主要支持按操作动作、操作对象类型、用户、关键词和时间范围查询日志。后续如需增加日志导出、异常告警、日志级别、IP 地址和客户端信息等字段，应通过扩展日志数据结构和查询接口实现。
4.7.3　 算法
　　本软件单元不涉及复杂数学算法，主要涉及日志写入算法、日志条件过滤算法、关键词模糊查询算法和日志排序限制算法。
a) 日志写入算法。系统接收数据库会话、当前用户、操作动作、操作对象类型、操作对象标识和操作说明；若当前用户不为空，则读取用户编号作为 user_id；若操作对象标识不为空，则统一转换为字符串；随后构造 AuditLog 对象，加入数据库会话并提交事务。
b) 条件过滤算法。系统接收管理员提交的 action、target_type、user_id、start_time、end_time 等查询条件；对于每一项非空条件，在基础日志查询对象上叠加对应过滤条件；未填写的条件不参与过滤。
c) 关键词模糊查询算法。若管理员输入 keyword，系统构造包含百分号通配符的匹配模式，并在 action、target_type、target_id 和 message 字段上进行或条件匹配，以提高日志检索的灵活性。
d) 排序限制算法。日志过滤完成后，系统按照 created_at 字段倒序排序，使最新日志优先显示；同时根据 limit 参数限制返回记录数量，避免单次查询返回过多数据。当前 limit 参数设置了最小值和最大值范围，用于控制查询规模。

4.7.7　 存储分配
　　日志审计软件单元主要涉及数据库存储、运行时查询对象和前端临时展示数据三类存储分配。
a) 数据库存储。审计日志持久化保存在 audit_logs 表中，表中保存操作用户、操作动作、操作对象类型、操作对象标识、操作说明和创建时间。该表是管理员查询系统操作历史的主要数据来源。
b) 运行时对象。日志查询过程中，后端会在内存中临时构造 AuditLog 查询对象，并根据查询条件逐步追加过滤条件。请求处理完成后，查询对象和返回列表由运行环境回收。
c) 前端展示数据。前端日志管理页面仅临时保存本次查询返回的日志列表和筛选条件，不作为持久化数据来源。刷新页面或重新查询时，应重新调用后端日志查询接口。
d) 关联存储。日志记录中的 user_id 可关联 users 表，target_id 与 target_type 共同指向被操作对象，但课程设计阶段不要求对所有操作对象建立强外键约束。
4.7.8　 限制条件
　　日志查询接口必须在用户登录后访问，且仅管理员角色用户可以访问。
　　日志写入应由后端业务软件单元调用，普通前端用户不得直接伪造日志写入请求。
　　日志记录应避免保存用户密码、访问令牌、数据库连接信息等敏感内容。
　　日志查询返回数量应受 limit 参数限制，避免一次性返回过多数据。
　　关键词检索仅面向 action、target_type、target_id 和 message 等文本字段，不对所有业务对象执行全文搜索。
　　当前课程设计阶段主要保留操作日志，不实现日志归档、日志清理、日志导出和异常告警功能。
　　日志记录用于辅助审计和问题定位，不应替代数据库事务、权限控制和业务数据一致性校验。
4.7.9　 出错设计
　　日志审计软件单元应按照错误类型返回明确提示，并避免因日志查询或写入异常导致系统整体不可用。

4.8　 文件存储软件单元 SYSMLDOCGEN-CSU-FILE
4.8.1　  功能描述
　　文件存储软件单元的项目唯一标识为 SYSMLDOCGEN-CSU-FILE。该软件单元是基于 SysML 模型的文档自动生成系统中负责非结构化文件保存、文件路径维护、导出文件生成和文件下载响应的基础业务单元，主要支撑模型管理与解析软件单元、文档生成与导出软件单元以及模板管理软件单元的文件处理需求。
　　本软件单元不直接承担模型语义解析、模板内容编辑或文档内容组织等业务规则，而是为上述业务过程提供统一的文件目录、文件命名、文件写入、文件读取、文件响应和关联文件清理能力。模型上传时，系统需要将用户提交的模型源文件保存至上传目录，并在模型记录中保存原始文件名和服务器存储路径；文档生成时，系统需要将生成后的 HTML 内容写入生成目录；文档导出时，系统需要按用户选择的格式生成 HTML、DOCX 或 PDF 文件，并通过文件响应返回给前端浏览器下载。
4.8.2　 非功能性/性能
　　文件存储软件单元应满足以下非功能性和性能要求：
a) 有效性要求。系统应能够正确创建上传目录和生成目录，并在文件写入成功后保存可追踪的文件路径。模型源文件、生成 HTML 文件以及导出的 DOCX、PDF 文件均应能够在指定目录下被定位和访问。
b) 效率要求。文件保存、文件导出和文件响应属于用户操作链路中的关键环节，应在普通课程设计演示环境下快速完成。对于体积较小的模型文件和工程文档，系统不应出现明显阻塞或页面长时间无响应。
c) 完整性要求。系统应同时保存文件实体和数据库元数据。模型源文件保存后，模型记录应保存 source_filename 和 stored_path；生成文档保存后，文档记录应保存 file_path、html_content、项目编号、模型编号和模板编号等关联信息。
d) 安全性要求。系统应避免直接使用用户上传文件名作为服务器端存储文件名，防止文件名冲突和路径注入风险。文件下载时应基于数据库中已授权的文档记录进行响应，不允许用户绕过权限直接读取任意服务器文件。
e) 可靠性要求。目录不存在时系统应自动创建目录；文件写入、导出或删除失败时应返回明确错误或进行容错处理。删除生成文档记录时，如文件物理删除失败，不应影响数据库记录删除结果，但应保留异常处理逻辑。
g) 可维护性要求。上传目录和生成目录应通过统一配置管理，不应在多个业务函数中分散硬编码。后续如需要迁移到服务器目录、网络文件系统或对象存储，应尽量通过配置和文件服务层进行扩展。
h) 可扩展性要求。当前系统主要支持模型源文件保存和 HTML、DOCX、PDF 三类文档导出。
4.8.3　 算法
　　本软件单元不涉及复杂数学算法，主要涉及文件存储路径解析算法、上传文件命名算法、生成文档写入算法、导出文件生成算法和关联文件清理算法。
a) 文件存储路径解析算法。系统从配置对象中读取 upload_dir 和 generated_dir 字段，并通过 upload_path 和 generated_path 属性转换为 Path 对象。业务单元调用时不直接拼接硬编码目录，而是通过配置属性获得目录路径。
b) 上传文件命名算法。系统读取上传文件原始文件名后提取文件后缀。若无法获得有效文件名，则采用默认后缀；随后使用 uuid4().hex 生成随机存储文件名，并将原始后缀拼接到随机文件名后，得到服务器端 stored_name。该处理方式可以降低同名文件覆盖风险。
c) 上传文件保存算法。系统首先根据 upload_path 创建上传目录，随后以二进制写入方式打开目标文件，并将 UploadFile 中的文件流复制到目标路径。写入成功后，系统将 stored_path 保存到模型记录中。
d) 生成文档写入算法。系统根据 generated_path 创建生成目录，将文档生成服务输出的 HTML 字符串写入文件，写入编码采用 UTF-8。写入后，系统将文件路径保存到 GeneratedDocument 记录的 file_path 字段。
e) 导出文件生成算法。用户请求导出时，系统读取生成文档记录，规范化文档标题作为文件名基础，根据导出格式选择处理分支。html 格式直接写入或读取 HTML 文件；docx 和 pdf 格式分别调用对应导出函数生成目标文件；生成成功后通过 FileResponse 返回给用户。
f) 关联文件清理算法。用户删除生成文档记录时，系统读取记录中的 file_path 字段，删除数据库记录并提交事务后，若文件路径存在且物理文件仍存在，则尝试删除该文件；若文件删除发生 OSError，系统进行容错处理，不影响主删除流程。

4.8.7　 存储分配
　　文件存储软件单元主要涉及服务器文件系统存储、数据库路径元数据存储和运行时临时对象三类存储分配。
a) 服务器文件系统存储。模型源文件保存在 upload_path 对应目录中，默认目录为 app/uploads；生成文档和导出文件保存在 generated_path 对应目录中，默认目录为 app/generated。系统在保存文件前调用 mkdir(parents=True, exist_ok=True) 创建目录。
b) 数据库元数据存储。模型源文件保存后，系统在 SysMLModel 记录中保存 source_filename、stored_path、version、uploaded_by、status 等信息；生成文档保存后，系统在 GeneratedDocument 记录中保存 file_path、html_content、project_id、model_id、template_id、created_by 等信息。
c) 运行时临时对象。文件处理过程中临时使用 UploadFile、Path、FileResponse、HTML 字符串、导出文件路径、数据库会话和当前用户对象。上述对象仅在请求处理期间存在，请求结束后由运行环境回收。
d) 文件命名分配。上传模型源文件使用 uuid4 随机值作为服务器端文件名主体，以减少文件名冲突；导出文档文件使用经安全化处理后的文档标题作为文件名主体，以便用户下载后识别。
e) 文件清理分配。删除生成文档记录时，系统尝试清理 file_path 指向的物理文件。当前阶段主要清理生成文档记录关联的 HTML 文件，DOCX 和 PDF 导出文件可由后续维护策略进行定期清理。
4.8.8　 限制条件
　　文件存储软件单元运行中受到以下限制：
　　文件存储目录必须具有后端服务进程的读写权限，否则模型上传、文档生成和导出操作可能失败。
　　模型源文件必须经模型管理与解析软件单元的权限校验后才能写入上传目录。
　　文档导出必须基于已存在且当前用户有权访问的生成文档记录，不允许用户直接指定服务器任意路径进行下载。
　　当前系统主要保存模型源文件、生成 HTML 文件以及按需导出的 DOCX、PDF 文件，不提供通用网盘式文件管理能力。
　　当前系统通过文件后缀保留上传文件类型，但不对所有文件内容进行深度安全扫描；后续真实部署时应增加文件大小限制、病毒扫描和更严格的 MIME 类型校验。
　　当前上传目录和生成目录通过本地文件系统实现，尚未接入对象存储、分布式文件系统或外部文件服务器。
　　删除生成文档记录时，物理文件清理采用尽力而为策略；若文件不存在或删除失败，不应导致系统主业务流程崩溃。
4.8.9　 出错设计
　　文件存储软件单元应针对路径配置、目录权限、文件写入、文件读取、导出格式和文件清理等异常情况进行处理，并尽量避免将服务器真实路径和后台异常堆栈直接暴露给前端用户。

4.8.10　 尚未解决的问题
　　当前文件存储软件单元已经能够满足课程设计阶段的模型源文件保存、生成文档保存、文档导出和文件下载要求，但仍存在以下后续可完善内容：
a) 当前文件存储主要基于本地文件系统，尚未接入对象存储、网络文件系统或独立文件服务器。
b) 当前系统尚未实现统一文件管理页面，用户不能单独查看所有上传文件和导出文件的存储状态。
c) 当前导出文件主要按需生成，尚未实现导出文件定期清理、过期策略和磁盘空间监控机制。
d) 当前上传文件安全校验以业务解析和后缀处理为主，尚未实现文件大小限制、病毒扫描、MIME 类型强校验和危险文件隔离。
e) 当前文件删除以生成文档记录关联文件为主，尚未对模型源文件删除、孤立文件清理和文件恢复进行完整设计。
f) 当前文件下载依赖后端 FileResponse，尚未实现断点续传、大文件分片下载和下载审计增强机制。
g) 当前文件存储功能可满足课程设计验收，但如后续扩展为真实工程系统，还需进一步完善文件安全、容量管理、备份恢复和长期归档机制。
